# 报告生成流程指导说明

这份说明是给 AI 工具使用的执行指南。

当用户希望生成报告时，只要同时提供：

- 本说明文件 `docs/process_guide.md`
- 本项目生成的 `outputs/ai_prompt_package.md`
- 用户补充的报告目标、场景、读者对象、输出格式要求

AI 就应按下述流程自动工作。

---

## 一、任务目标

你需要基于用户提供的参考资料，生成一份**可行、可信、有依据**的报告。

你的基本原则：

1. 只基于资料中的证据进行表述。
2. 无证据支撑时，明确说明资料不足。
3. 每个重要结论尽量附带来源文件名。
4. 区分客观事实、分析判断、推断假设。
5. 遇到矛盾信息时，不强行统一，先列冲突点。
6. 如用户允许联网搜索，联网结果只能作为补充线索，必须二次核验后才能写入报告。

---

## 二、标准执行流程

### 第 1 步：理解任务

先识别以下信息：

- 报告主题是什么
- 读者是谁
- 报告目标是什么
- 希望输出的形式是什么（Markdown、PPT 提纲、正式公文、商业分析等）
- 是否有字数、结构、语气限制

如果用户未明确说明，可先按“专业、审慎、面向决策”的报告标准执行。

### 第 2 步：检查资料处理状态

阅读 `outputs/ai_prompt_package.md` 中的：

- 资料清单
- 证据摘录
- 可用 skills
- 质量检查清单

先判断：

- 哪些文件成功读取
- 哪些文件读取失败或存在告警
- 证据是否足以支持正式报告

### 第 2.5 步：判断是否需要联网补充

仅在同时满足以下条件时，才允许使用联网搜索：

- 本地资料不足以回答关键问题；
- 用户明确允许联网；
- 当前环境已配置联网搜索接口；
- 能够找到权威来源进行二次核验。

如果不满足上述条件，应继续基于本地资料写作，并明确说明资料边界。

### 第 3 步：建立证据视图

把已提取内容整理成以下四类：

1. **事实信息**：明确陈述、数字、时间、对象、事件。
2. **分析线索**：趋势、对比、因果、优势、风险。
3. **决策依据**：可支持建议或结论的证据。
4. **待确认项**：缺失信息、冲突信息、低置信度信息。

### 第 4 步：先出大纲再写正文

推荐先生成一个报告大纲，常用结构如下：

1. 执行摘要
2. 背景与目标
3. 核心发现
4. 证据与分析
5. 风险与限制
6. 建议与下一步行动
7. 待确认事项
8. 附录/引用来源

### 第 5 步：正文写作规则

写作时必须遵守：

- 关键结论后附来源文件名。
- 避免“显然”“一定”“毫无疑问”等绝对化措辞。
- 对推断内容使用：
  - “初步判断”
  - “可能”
  - “依据当前资料推测”
  - “仍需进一步核实”
- 如果某项建议缺乏充分数据，应改写为条件性建议。

### 第 6 步：引用与溯源规则

优先使用以下引用形式：

- `来源：xxx.docx`
- `依据：xxx.pdf 第 3 页`
- `参考：xxx.xlsx 工作表“预算测算”`
- `摘录自：xxx.pptx 第 5 页幻灯片`
- `联网核验来源：某政府官网/标准发布页，检索时间：YYYY-MM-DD`

如果使用了联网搜索，必须遵守：

1. 先记录检索词和检索时间；
2. 只把搜索结果作为线索，不直接当作最终证据；
3. 对关键事实回到原始权威页面复核；
4. 若只能找到新闻转载、博客、论坛或二手摘要，则不得写成确定事实。
5. 对候选链接优先调用网页正文抓取能力，再判断是否纳入证据。

### 第 7 步：自检与修正

在输出最终报告前，逐项检查：

- 是否有无依据的结论
- 是否遗漏明显重要资料
- 是否把猜测写成事实
- 是否对矛盾信息进行了提示
- 是否列出风险与待确认事项

## 三、Skills 使用说明

当需要理解资料时，按文件类型调用对应 skill：

- `.txt` / `.md` → `text_reader`
- `.pdf` → `pdf_reader`
- `.docx` → `docx_reader`
- `.pptx` → `pptx_reader`
- `.xlsx` → `xlsx_reader`
- 联网公开检索 → `web_search`（仅在用户允许且环境已启用时可用）
- 网页正文抓取 → `fetch_web_page`（仅在用户允许且环境已启用时可用）

如果某个文件读取失败：

1. 不要忽略它。
2. 在报告中注明该资料未能成功解析。
3. 评估这是否影响结论可靠性。
4. 如影响明显，在“待确认项”里单列。

如果调用 `web_search`：

1. 先检索公开线索；
2. 再访问权威原始来源做二次核验；
3. 记录最终采用来源、检索时间、未采纳原因；
4. 对未完成核验的信息，放入“待确认事项”。
5. 对拟采用页面，优先使用 `fetch_web_page` 抽取正文，避免只凭搜索摘要写作。

---

## 四、建议的输出格式

建议最终输出包含以下部分：

### 1. 执行摘要
- 用 3~6 条概括结论。

### 2. 核心发现
- 每条发现都应附依据。

### 3. 详细分析
- 分主题展开。
- 对比不同资料时标注来源。

### 4. 风险与限制
- 列出资料不足、解析限制、推断风险。

### 5. 行动建议
- 区分短期、中期、长期建议。
- 每项建议说明触发条件或前提。

### 6. 待确认事项
- 单独列清单，方便后续补料。

---

## 五、禁止事项

- 禁止编造数据、访谈、结论、案例。
- 禁止把推测写成已证实事实。
- 禁止忽略资料冲突。
- 禁止为了报告完整性而填补不存在的信息。
- 禁止把未经核验的联网搜索结果直接写入正式结论。

---

## 六、推荐执行口令

如果用户没有额外指令，可按以下思路执行：

> 请先阅读资料清单与证据摘录，建立事实、判断、待确认项三类信息表；再输出报告大纲；最后生成正式报告，并确保每个重要结论都有来源依据，所有证据不足之处必须明确说明。
